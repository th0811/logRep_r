# LogRep2 改修指示書

## 1. 文書の目的

本書は、既存のログ収集アプリ `logRep_r` とログ分析アプリ `logAnalyzer` を、新アプリ `LogRep2` に統合するための実装指示書である。

開発、レビュー、テスト、受け入れ確認では本書を基準とする。実装中に本書と異なる設計が必要になった場合は、独断で大きく変更せず、理由、影響範囲、代替案を整理して承認を得たうえで本書を更新すること。

## 2. 改修の目的

次の目的を達成する。

- `logRep_r` と `logAnalyzer` を一つのWindowsアプリに統合する。
- ログ入力先、ログ出力先など、両アプリで共通する設定を一本化する。
- 既存のログ収集、正規化、過去ログ分析機能を維持する。
- 収集中のログを対象としたリアルタイム分析を追加する。
- 分析結果を任意位置へ表示できる透過オーバーレイを追加する。
- 既存のセッション出力形式とHTMLログ再現ツールとの互換性を維持する。

統合後のアプリ名、実行ファイル名は次のとおりとする。

```text
アプリ名: LogRep2
実行ファイル名: LogRep2.exe
```

## 3. 開発場所

すべての新規開発は次のフォルダー以下で行う。

```text
logRep2/
```

既存の次のフォルダーは、移植元、比較元、回帰確認対象として当面維持する。

```text
logRep_r/
logAnalyzer/
AssistantTool/
```

既存アプリへ直接大規模な変更を加えず、必要な機能を `logRep2` へ段階的に移植すること。既存アプリを削除する作業は本改修の対象外とする。

## 4. 対象範囲

### 4.1 統合対象

- FFXI TEMPログ監視
- ログ読込、デコード、時刻抽出、マーカー検出
- rawレコード生成、重複排除、保存
- canonicalレコード生成、統合、重複排除、保存
- セッションの開始、終了、異常終了管理
- 状態、統計情報の保存
- 過去セッションの選択、範囲指定、分析
- actor別、action別など、既存の分析結果表示
- PC名、NPC名の登録と分類
- タスクトレイ、単一起動制御
- 無理のない範囲でのCLI機能
- 統合設定画面

### 4.2 新規機能

- リアルタイム分析
- リアルタイム分析の途中リセット
- 分析結果オーバーレイ
- オーバーレイの表示位置、サイズ、透過度などの保存
- 旧設定から統合設定への初回移行
- リアルタイム分析の性能およびメモリ診断情報

### 4.3 対象外

- `AssistantTool` の統合
- `AssistantTool` の全面改修
- 旧アプリ名のexeを利用した互換ランチャー
- 旧CLIとの完全互換
- オーバーレイのクリック透過
- 排他的フルスクリーンでのオーバーレイ動作保証
- DirectXへの描画挿入
- 初版での差分集計
- ユーザーホーム、`AppData`、レジストリへの設定保存

## 5. 対応環境

- Windows 10 64bit
- Windows 11 64bit
- .NET 8
- WPF
- x64を正式な配布対象とする。

オーバーレイは、FFXIのウィンドウモードおよび枠なしウィンドウモードを対象とする。排他的フルスクリーンは動作保証対象外であることをREADMEへ明記すること。

## 6. 推奨ソリューション構成

初期構成は次を基本とする。実装前の依存関係調査により小さな調整は可能だが、責務を一つのWPFプロジェクトへ集中させないこと。

```text
logRep2/
├─ LogRep2.sln
├─ Directory.Build.props
├─ global.json
├─ README.md
├─ config.example.json
├─ LogRep2_改修指示書.md
├─ src/
│  ├─ LogRep2.App/
│  ├─ LogRep2.Collection/
│  ├─ LogRep2.Analysis/
│  ├─ LogRep2.Infrastructure/
│  └─ LogRep2.Ipc/
└─ tests/
   └─ LogRep2.Tests/
```

各プロジェクトの責務は次のとおりとする。

| プロジェクト | 責務 |
| --- | --- |
| `LogRep2.App` | WPF画面、ViewModel、トレイ、アプリケーションライフサイクル |
| `LogRep2.Collection` | TEMP監視、デコード、正規化、重複排除、セッション管理 |
| `LogRep2.Analysis` | 過去ログ分析、リアルタイム集計、分析ルール |
| `LogRep2.Infrastructure` | 統合設定、設定移行、ファイル保存、診断 |
| `LogRep2.Ipc` | 単一起動、必要な範囲のCLI・プロセス間通信 |

依存方向は、Appから各機能へ向かう一方向を基本とする。CollectionとAnalysisが互いのWPF型へ依存してはならない。

## 7. 既存コードの移植方針

### 7.1 基本方針

- 既存コードの意図と動作を維持する。
- 一度に全面書き換えせず、ビルド可能な小さな単位で移植する。
- 既存のCore処理を優先して再利用する。
- 移植と新機能追加を同時に進めず、まず既存機能の統合を完了させる。
- 名前空間は新規コードから `LogRep2.*` へ整理する。
- コメント、ログ、画面メッセージ、例外メッセージは日本語に統一する。

### 7.2 共通モデル

両アプリに存在するcanonicalレコード、セッション情報、状態などの類似モデルを調査すること。

モデルを統合する場合は、既存JSONのフィールド名、型、null許容、時刻表現、schema versionへの影響を確認する。単に同名であることを理由に統合してはならない。

### 7.3 既存テスト資産

現状の作業ツリーでは、既存ソリューションに登録されたテストプロジェクトのソースおよび `.csproj` が存在せず、`bin` と `obj` だけが残っている。実装開始時に履歴または別保管場所を確認し、復元できない場合は、既存仕様書とサンプルログを基準としてLogRep2側に回帰テストを再作成すること。

## 8. セッション出力の互換性

LogRep2は、少なくとも次の既存ファイルを従来どおりセッションフォルダーへ出力する。

```text
session.json
raw_records.jsonl
canonical_records.jsonl
state.json
stats.json
```

次のルールを守ること。

- 既存フィールドを削除、改名しない。
- 既存フィールドの意味や型を変更しない。
- 新規フィールドは後方互換を保った追加とする。
- スキーマ変更時はschema versionを適切に更新する。
- 旧`logRep_r`で作成したセッションをLogRep2で分析できること。
- LogRep2で作成したセッションを既存`AssistantTool`で利用できること。

`AssistantTool`は独立ツールとして現行の使い方を維持する。必要な作業は互換性テストに限定し、明示的な指示がない限りコードを変更しないこと。

## 9. 統合設定

### 9.1 保存場所

設定はポータブル方式とし、デフォルトでは `LogRep2.exe` と同じフォルダーへ保存する。

```text
LogRep2.exe
LogRep2.settings.json
```

次の場所へ設定を作成してはならない。

- ユーザーホームディレクトリ以下
- `%LOCALAPPDATA%`
- `%APPDATA%`
- レジストリ

exeのあるフォルダーへ書き込めない場合、別の保存場所へ暗黙に切り替えてはならない。保存できなかったこと、対象パス、利用者が取るべき対応を日本語で表示すること。

CLIの明示的な設定ファイル指定を残す場合は、`--config`で指定されたパスを最優先としてよい。

### 9.2 設定ファイル

ファイル名は次を基本とする。

```text
LogRep2.settings.json
```

設定モデルは機能別にグループ化し、設定スキーマのバージョンを持たせる。

```json
{
  "schema_version": 1,
  "collection": {
    "temp_directory": "",
    "output_directory": "sessions",
    "polling_interval_ms": 1000
  },
  "analysis": {
    "known_pc_names": [],
    "known_npc_names": [],
    "realtime_refresh_interval_ms": 500
  },
  "overlay": {
    "enabled": false,
    "opacity": 0.8,
    "topmost": true,
    "position_locked": true
  },
  "application": {
    "auto_start_collection_on_launch": false,
    "minimize_to_tray_while_collecting": false
  }
}
```

上記は構造例であり、既存設定項目を失わないよう、実装時に `CollectorConfig` と `AnalyzerSettings` の全項目を棚卸しすること。

### 9.3 設定値の一本化

既存分析側のセッションルート設定は廃止し、統合後は収集設定の出力フォルダーを過去ログ分析のデフォルトルートとして使用する。

過去ログ分析で一時的に別フォルダーを開く機能は残してよい。ただし、その操作によって収集出力先が意図せず変更されないようにすること。

### 9.4 旧設定の移行

LogRep2の初回起動時に、`LogRep2.exe` と同じフォルダーから次の旧設定を検出する。

```text
config.json
analyzer_settings.json
```

利用者には、LogRep2を旧exeと同じフォルダーへ配置する方法を推奨する。

移行ルールは次のとおりとする。

- `config.json`から収集設定を移行する。
- `analyzer_settings.json`からPC名、NPC名などの分析設定を移行する。
- セッション出力先は`config.json`の出力先を優先する。
- 旧設定ファイルを削除、上書きしない。
- 移行結果と警告を日本語で表示または記録する。
- 移行済みであることを新設定のスキーマ情報で判定する。
- 一部の値が不正でも、可能な項目は移行し、不正項目と採用した初期値を通知する。
- 新設定が既に存在する場合、自動移行で上書きしない。

## 10. 画面構成

メイン画面から少なくとも次の機能へ移動できるようにする。

- 収集状態
- リアルタイム分析
- 過去ログ分析
- 設定
- オーバーレイ表示切替

既存機能を見失わないよう、移植初期は既存画面構成を大きく作り直さず、統合後に必要最小限のナビゲーションを追加すること。

収集の「開始・停止」と、リアルタイム分析の「開始・終了・リセット」は、内部状態を分離すること。利用者が混同しない文言と状態表示を用意する。

## 11. リアルタイム分析

### 11.1 分析範囲

リアルタイム分析範囲は、利用者が「分析開始」ボタンを押してから「分析終了」ボタンを押すまでとする。

- 分析開始前のログは初期状態では対象外とする。
- 分析開始時点のcanonical位置を開始位置として記録する。
- 分析終了時点のcanonical位置を終了位置として固定する。
- 分析終了後に追加されたログを最終結果へ反映しない。
- 収集を停止しても、分析終了状態と結果を明確に管理する。

ボタン名は、収集操作との混同を防ぐため、画面上では次を基本とする。

```text
分析開始
分析終了
分析リセット
```

### 11.2 分析リセット

分析中にリセットできること。

- その時点までのリアルタイム分析結果をクリアする。
- リセット時点のcanonical位置を新しい分析開始位置とする。
- 収集処理は停止しない。
- リアルタイム分析は継続する。
- ディスクへ保存済みのセッションログは削除しない。
- リセット前のデータは、後から過去ログ分析できること。

### 11.3 データ連携

同一プロセス内で、収集・正規化処理から分析処理へ更新を通知する。リアルタイム分析のために `canonical_records.jsonl` を周期的に再読込する方式を主方式にしてはならない。

基本フローは次のとおりとする。

```text
TEMPログ
  → 収集、デコード、正規化
  → canonicalスナップショット更新
       ├→ 既存形式でディスクへ保存
       └→ リアルタイム分析へ更新通知
              → 集計
              → 通常画面とオーバーレイを更新
```

ファイル保存の失敗が直ちに分析画面の停止へ波及しないよう、保存処理とリアルタイム表示の責務を分離すること。ただし、保存失敗は利用者へ明確に警告する。

### 11.4 初版の集計方式

初版は、メモリ上のcanonicalスナップショットを用いた全件再集計方式とする。

ここでいう全件とは、現在のリアルタイム分析開始位置または直近のリセット位置から、最新位置までの分析対象全件を意味する。

- ディスク上のJSONLを毎回読み直さない。
- 収集側が保持するcanonicalレコードを読み取り専用スナップショットとして共有する。
- 分析側でレコード本体を丸ごと複製しない。
- 分析範囲は可能な限りインデックスまたは不変の位置情報で表現する。
- 複数の再集計を同時実行しない。
- 新しい更新要求が来た場合、古い再集計をキャンセルまたは破棄し、最新結果を優先する。
- 250～1,000msの範囲で更新をまとめる。初期値は500msを推奨する。
- 集計はUIスレッドを長時間占有しない。
- UIへ渡すのは主に集計結果とし、全レコードの複製を避ける。

初版では差分集計を実装しない。性能測定によって必要性が確認された場合に限り、後続改修として検討する。

### 11.5 結果の整合性

同一の分析範囲、分析ルール、PC/NPC登録情報を使用した場合、リアルタイム分析の最終結果と保存済みセッションの再分析結果が一致すること。

## 12. メモリおよび性能

### 12.1 メモリ方針

メモリ上のスナップショットを利用するため、レコード件数に応じてメモリ使用量が増加する。次の対策を必須とする。

- x64で動作させる。
- canonicalレコード本体を収集側と分析側で二重保持しない。
- リセット後に不要となった分析用参照や一時結果を解放する。
- 再集計ごとの一時オブジェクトを長期保持しない。
- イベント購読解除、CancellationToken、タイマー停止を適切に行う。
- オーバーレイ非表示時に不要な描画更新を続けない。

### 12.2 初版目標

初版の性能目標は次を基準とし、実装後の実測結果を記録する。

- canonical 10,000件のリアルタイム分析時に、LogRep2全体の使用メモリがおおむね300MB以内であること。
- 分析開始、リセット、再集計を繰り返しても、使用メモリが継続的に増え続けないこと。
- 通常の更新で画面操作が著しく停止しないこと。
- 500ms更新設定に対し、処理が追いつかない場合でも再集計を無制限にキューへ蓄積しないこと。

目標値を超えた場合は、レコード件数、処理時間、一時メモリ、原因を測定し、差分集計へ直ちに変更する前に二重保持や不要な割り当てを解消すること。

### 12.3 診断情報

デバッグ表示または診断ログで、少なくとも次を確認可能にする。

- canonical総件数
- 現在の分析対象件数
- 最新の再集計時間
- 再集計のキャンセルまたは破棄回数
- プロセス使用メモリ
- 最終更新日時

利用者向けログと開発診断ログの粒度は分離してよい。

## 13. オーバーレイ

### 13.1 基本仕様

オーバーレイはWPFの独立ウィンドウとして実装する。

- 常に最前面へ表示できること。
- 背景の透過度を変更できること。
- 画面上の任意位置へ移動できること。
- サイズを変更できること。
- 位置とサイズを固定できること。
- 表示、非表示を切り替えられること。
- 通常画面またはタスクトレイから再表示できること。
- 初版ではクリック透過を実装しない。

### 13.2 初版設定項目

- 表示有無
- 常に最前面
- 透過度
- 左位置、上位置
- 幅、高さ
- 位置固定
- 表示対象モニターを復元するための情報
- フォントサイズ
- 表示行数
- 表示項目

初版の表示項目は既存分析結果から安定して取得できるものを優先し、少なくとも次を候補とする。

- 合計ダメージ
- DPSまたはDPS状態
- 命中率
- actor別順位
- リアルタイム分析状態

詳細な初期表示項目はUI実装前に既存分析結果との対応を確認し、テスト可能な範囲に絞ること。

### 13.3 操作性と安全対策

- 編集モードではドラッグ移動とサイズ変更を可能にする。
- 固定モードでは意図しない移動を防ぐ。
- 透明度には操作不能にならない最低値を設定する。
- 「位置をリセット」を用意する。
- モニターが切断された場合、プライマリーモニターの表示領域内へ戻す。
- WindowsのDPI倍率が変わっても画面外へ復元しない。
- オーバーレイの例外で収集処理やセッション保存を停止させない。

### 13.4 非対応事項

排他的フルスクリーンでは、通常のWPF最前面ウィンドウがゲーム上へ表示されない場合がある。初版では動作保証せず、DirectX描画挿入などの特殊対応を行わない。

## 14. CLIおよびIPC

旧CLIの全コマンドを完全維持する必要はない。ただし、実装負担が過大でない範囲で引き続き利用可能にする。

優先度は次のとおりとする。

1. ヘルプ
2. 収集開始
3. 収集停止
4. 状態確認
5. 一度だけの収集
6. 設定パス表示
7. 設定値の取得、変更

コマンド名、引数、出力形式は整理のため変更してよい。変更内容はREADMEに記載する。GUI起動中のCLI操作を継続する場合は、単一起動制御およびIPCとの整合性を確保する。

リアルタイム分析操作のCLI対応は初版の必須要件としない。

## 15. エラー処理とログ

- コメント、ログ、例外メッセージ、利用者向けメッセージは日本語とする。
- ファイルパスを伴うエラーでは、可能な範囲で対象パスを示す。
- 設定読込失敗時に無言で全項目を初期化しない。
- 収集、保存、分析、オーバーレイのエラーを区別する。
- オーバーレイまたは分析表示の障害が収集データを失わせないようにする。
- 収集異常終了時は既存の`aborted`相当の状態管理を維持する。
- ログへ機密情報や不要な大量データを出力しない。

## 16. 実装フェーズ

### フェーズ0: 現状基準の確立

- 既存機能一覧を作成する。
- 既存設定項目を全件棚卸しする。
- 重複モデルとスキーマ差異を調査する。
- サンプルログから基準となる出力ファイルと分析結果を用意する。
- 既存テスト資産の復元可否を確認する。
- 文字化けして見える既存文言について、ファイル文字コードと実際の画面表示を確認する。

### フェーズ1: ソリューション構築と既存Core移植

- `LogRep2.sln`と各プロジェクトを作成する。
- 収集Coreを移植する。
- 分析Coreを移植する。
- 既存サンプルで出力と分析結果を比較する。
- この段階では新機能による既存ロジックの変更を最小限にする。

### フェーズ2: UI統合

- LogRep2のアプリケーションライフサイクルを構築する。
- 収集画面と過去ログ分析画面を一つのアプリへ統合する。
- タスクトレイと単一起動制御を統合する。
- 既存機能をLogRep2単体で操作可能にする。

### フェーズ3: 設定一本化

- 統合設定モデルと保存処理を作成する。

## 17. テスト方針

### 17.1 単体テスト

- 設定の読込、保存、検証
- 旧設定からの移行
- パス解決
- TEMPログファイル名解析
- デコード、時刻抽出、マーカー検出
- raw、canonical重複排除
- セッション状態遷移
- 分析範囲
- actor分類
- 集計、DPS、命中率などの既存分析
- 分析開始、終了、リセット状態遷移
- 更新要求のキャンセルと最新結果優先
- オーバーレイ位置の復元と画面内補正

### 17.2 統合テスト

- サンプルTEMPログからセッション一式を生成できる。
- 生成したセッションを過去ログ分析できる。
- 収集しながらリアルタイム分析できる。
- リセット後はリセット以降だけが集計される。
- 分析終了後のログが結果へ入らない。
- リアルタイム最終結果と保存後の再分析結果が一致する。
- 旧設定を新設定へ移行できる。
- 旧セッションを読み込める。
- LogRep2出力を既存`AssistantTool`で利用できる。

### 17.3 長時間・性能テスト

- canonical 10,000件以上で操作性とメモリを測定する。
- 分析開始、リセット、終了を繰り返す。
- オーバーレイ表示、非表示を繰り返す。
- 収集停止、再開を繰り返す。
- 再集計要求が連続しても処理待ちが無制限に増えないことを確認する。
- ガベージコレクション後も不要なデータが参照され続けていないことを確認する。

### 17.4 Windows表示テスト

- Windows 10、Windows 11
- 100%、125%、150%以上のDPI
- 単一モニター
- 複数モニター
- セカンダリーモニター切断後の復元
- ウィンドウモードおよび枠なしウィンドウモード

排他的フルスクリーンは参考確認に留め、合否条件に含めない。

## 18. 受け入れ条件

次の条件をすべて満たすこと。

### 18.1 統合

- `LogRep2.exe`一つでログ収集とログ分析を操作できる。
- 既存の主要な収集機能が維持されている。
- 既存の主要な過去ログ分析機能が維持されている。
- 旧exeや互換ランチャーを必要としない。

### 18.2 設定

- 設定が`LogRep2.exe`と同じフォルダーへ保存される。
- ユーザーホーム、`AppData`、レジストリへ設定を作成しない。
- 書き込み不能時に別フォルダーへ暗黙に保存しない。
- exeと同じフォルダーの旧設定を検出し、移行できる。
- 旧設定を削除、上書きしない。
- 収集出力先と分析のデフォルトセッションルートが一本化されている。

### 18.3 リアルタイム分析

- 「分析開始」から「分析終了」までを分析できる。
- 分析中に「分析リセット」を実行できる。
- リセットしても収集やディスク保存が継続する。
- 初版はメモリ上のスナップショット全件再集計で動作する。
- canonicalレコード本体を収集と分析で不必要に二重保持しない。
- 最終結果が同範囲の過去ログ再分析結果と一致する。
- 更新中も画面が実用上操作可能である。

### 18.4 オーバーレイ

- 任意位置へ移動できる。
- サイズ、透過度を設定できる。
- 常に最前面表示を設定できる。
- 位置を固定できる。
- 設定が再起動後も保持される。
- モニター構成変更後も画面外へ取り残されない。
- オーバーレイ障害が収集処理を停止させない。
- クリック透過を実装していない。
- 排他的フルスクリーンを保証対象としていないことが文書化されている。

### 18.5 互換性

- 旧`logRep_r`のセッションをLogRep2で分析できる。
- LogRep2の出力形式が既存仕様と互換である。
- LogRep2出力を既存`AssistantTool`で利用できる。

### 18.6 品質

- ビルドが成功する。
- 追加したテストが成功する。
- 警告を可能な限り解消している。
- コメント、ログ、例外メッセージが日本語で統一されている。
- canonical 10,000件時の使用メモリがおおむね300MB以内である。
- 操作を繰り返してもメモリ使用量が継続的に増え続けない。

## 19. 実装中の判断ルール
