# LogRep2 フェーズ4 作業報告書

## 1. 作業概要

フェーズ4として、収集処理が保持するcanonicalレコードを同一プロセス内でリアルタイム分析へ共有し、メイン画面から分析開始、分析終了、分析リセットを操作できる機能を実装した。

初版の集計方式は改修指示書どおり「メモリ上のスナップショット全件再集計」とした。差分集計、オーバーレイ、クリック透過は実装していない。

## 2. 実装内容

### 2.1 読み取り専用canonical共有境界

`LogRep2.Contracts`プロジェクトを追加し、`ICanonicalRecord`を収集処理と分析処理の共通境界とした。

- CollectionとAnalysisのどちらからもWPF型へ依存しない。
- 収集側と過去ログ分析側の既存canonicalモデルはJSON仕様を維持したまま共通インターフェースを実装する。
- リアルタイム分析には収集側canonicalレコードと同じオブジェクト参照を渡す。
- 分析範囲の配列は参照だけを保持し、canonicalレコード本体を複製しない。
- 既存の`canonical_records.jsonl`形式および過去ログ読込モデルは変更していない。

収集ポーリングごとに、セッションID、更新日時、読み取り専用レコード一覧を持つ`CanonicalSnapshot`を通知する。リアルタイム分析の主経路ではJSONLを再読込しない。

### 2.2 分析状態管理

`RealtimeAnalysisController`に次の独立状態を実装した。

| 状態 | 説明 |
| --- | --- |
| `Stopped` | 初期状態。分析開始前 |
| `Running` | 分析開始から分析終了まで |
| `Completed` | 分析終了位置を固定し、最終結果を保持 |

- 分析開始時のsnapshot件数を開始インデックスとして記録し、開始前のレコードを除外する。
- 分析終了時の件数を終了インデックスとして固定し、終了後の更新を最終結果へ反映しない。
- 分析リセット時は現在件数を新しい開始インデックスとし、結果を消去して`Running`を維持する。
- リセットは収集処理および保存済みファイルに影響しない。
- 分析中に新しい収集セッションへ切り替わった場合、新セッションの先頭を開始位置とする。

### 2.3 再集計、デバウンス、最新結果優先

`RealtimeAnalysisEngine`は既存の分析ロジックを再利用し、対象インデックス範囲を全件再集計する。更新間隔には統合設定の`analysis.realtime_refresh_interval_ms`を使用し、実行時は250～1,000msへ制限する。初期値は500msである。

- 新しい更新要求が来た場合、待機中の古い要求をキャンセルする。
- 実行中の古い結果は世代番号で判定して破棄する。
- 複数要求を無制限にキューへ蓄積しない。
- 集計は`Task.Run`で実行し、UIスレッドを長時間占有しない。
- 終了、リセット、アプリ終了時にCancellationTokenSourceとイベント購読を解放する。

### 2.4 メイン画面

メイン画面に「リアルタイム分析」領域を追加した。

操作項目:

- 分析開始
- 分析終了
- 分析リセット

表示項目:

- 分析状態
- canonical総件数と現在の分析対象件数
- 合計ダメージと合計DPS
- 最新の再集計時間
- 再集計の破棄・取消回数
- プロセスWorking Set
- 最終更新日時

収集開始・停止とは別のコマンドおよび状態として実装した。

## 3. テスト追加

- 指定インデックス範囲だけを集計すること。
- 同じ範囲を既存の通常分析経路で集計した結果と、actor、action、分析時間が一致すること。
- canonical 10,000件を全件再集計できること。
- 分析開始前のレコードが対象外になること。
- リセット後も分析中状態を維持し、リセット後のレコードだけを対象にすること。
- 分析終了後のレコードが最終結果へ入らないこと。
- 更新要求集中時に古い要求を破棄して最新snapshotを採用すること。

## 4. 性能測定

2026-07-17にDebug構成の自動テストで、合成canonicalレコード10,000件を1回全件再集計した。

| 項目 | 実測値 |
| --- | ---: |
| 対象件数 | 10,000件 |
| 再集計時間 | 402.2ms |
| 再集計中のスレッド割当量 | 45.0MB |
| テストプロセスWorking Set | 70.2MB |

値は開発PC上の合成データとテストホストによる参考値であり、実ログ内容、他機能の使用状況、PC性能によって変動する。Working Setは初版目標の300MB未満だった。

分析側が保持する範囲配列はレコード参照のみで、canonicalレコード本体の二重保持は行っていない。全件再集計時には解析中間オブジェクトの一時割当が発生するが、結果更新後に長期保持しない。

## 6. 変更理由と影響範囲

変更理由:

- JSONLの周期的再読込を避け、収集直後のデータを低遅延で分析するため。
- canonicalレコード本体の二重保持を避けるため。
- 収集操作とリアルタイム分析操作を独立させるため。
- 要求集中時のUI停止やキュー増大を防ぐため。

影響範囲:

- `LogRep2.Contracts`: canonical共有契約を追加。
- `LogRep2.Collection`: snapshot通知を追加。
- `LogRep2.Analysis`: 共通契約の受入れとリアルタイム集計エンジンを追加。
- `LogRep2.App`: 状態管理、メイン画面操作、結果・診断表示を追加。
- 既存`logRep_r`、`logAnalyzer`、`AssistantTool`は変更していない。
- セッション出力JSONのフィールド、型、schema versionは変更していない。

## 7. 制限事項

- 初版仕様どおり差分集計は実装していない。
- 合計DPSはDPSを算出できたactorのDPS合計として表示する。分析時間が確定できない場合は`-`を表示する。
- 実ゲームを接続した長時間連続収集、リセット反復、セッション切替の手動耐久確認は受け入れ確認で実施する。
- オーバーレイはフェーズ5対象であり未実装。
